## Group 1 — Ready to use, no cleaning required

### FineWeb-Edu

**Used in:** Both plans (pretraining — 71% budget / 51% full)

HuggingFace's own team built and cleaned this dataset before releasing it. The cleaning pipeline includes:

- HTML → plain text via Trafilatura (high-quality web scraper)
- MinHash LSH deduplication across the full Common Crawl
- Language detection — English only
- Educational content scoring by a trained LLM classifier
- URL-based filtering (adult content, spam, low-quality domains removed)

**What you get from HuggingFace:** Clean UTF-8 text in Parquet files. No markup, no duplicates, no garbage.

**Action required:** None. Stream directly into the tokenizer.

---

### FinCoT

**Used in:** Both plans (SFT)

Synthetic dataset generated by GPT-4o. 9.2K chain-of-thought reasoning examples over financial QA. High-quality output — GPT-4o does not produce HTML artifacts, formatting noise, or structural inconsistencies.

**Action required:** None. Convert to our JSONL conversation format and use directly.

---

### SmolTalk

**Used in:** Both plans (SFT — existing dataset we already use)

Already working in our current SFT pipeline. Clean, high-quality synthetic instruction-response pairs from HuggingFaceTB.

**Action required:** None. We have this running.

## Group 2 — Near-ready, minimal work (format extraction only)

These datasets have clean text but require pulling the right fields from their HuggingFace schema before passing to the tokenizer or SFT pipeline.

---

### OpenHermes 2.5

**Used in:** Both plans (pretraining — 3%)

Synthetic reasoning dataset generated by GPT-4/Nous Hermes. Clean text. The only work is extracting the right fields: each row is a conversation with `system`, `human`, `gpt` fields. For pretraining we concatenate them into a plain text stream:

```
<|system|> You are a helpful assistant. <|human|> What is ... <|gpt|> ...
```

No HTML, no artifacts. The text itself is high quality.

**Estimated effort:** 20–30 lines of extraction code.

---

### Finance-Alpaca

**Used in:** Budget plan (SFT)

68K financial QA pairs, MIT licensed. Generated by GPT-3.5 (older model) plus existing financial QA sources. The text quality is good but the schema uses `instruction` / `input` / `output` fields rather than our `role` / `content` conversation format.

**What needs doing:**
- Map `instruction` + `input` → `user` message
- Map `output` → `assistant` message
- Write to our JSONL format

**Estimated effort:** ~20 lines of conversion code in `prepare_finance_sft.py`.

One minor quality issue: GPT-3.5-era generation occasionally produces short or slightly vague answers. At 68K rows this is acceptable — the quantity and topic diversity outweigh occasional quality variance.

---

### Reuters Financial News

**Used in:** Full plan only (pretraining — ~0.2%)

105K professional Reuters news articles, 2006–2013. Journalism is inherently clean: no HTML, no markup, structured prose. The dataset has `headline`, `summary`, and `body` fields in JSON.

**What needs doing:**
- Concatenate `headline + "\n" + body` per article for the pretraining stream
- Skip `summary` (it duplicates the body content, leading to near-duplicate text)

**Estimated effort:** ~15 lines of field extraction code.

## Group 3 — Light cleaning required

### EDGAR-Corpus

**Used in:** Full plan only (pretraining — 10%)

220K annual SEC filings, pre-split into 15 standardised sections (Item 1, Item 1A Risk Factors, Item 7 MD&A, etc.). The section-splitting is the main cleaning work already done. What remains:

**Issues to fix:**
1. **Item headers left in text** — lines like `"ITEM 7. MANAGEMENT'S DISCUSSION AND ANALYSIS OF FINANCIAL CONDITION"` appear at the start of sections. They add no information for language modelling.
2. **Page number artifacts** — occasional `"F-12"` or `"Page 47 of 112"` remnants from PDF parsing.
3. **Empty sections** — some companies have empty items (e.g. Item 5 with just "None"). These should be skipped.

**What does NOT need fixing:**
- Text is already plain text — no HTML or XBRL
- Sections are already labelled and separated — no need to parse document structure
- Financial tables are in text form but that is acceptable (numbers train the model on financial magnitudes)

**Estimated effort:** A 50–80 line cleaning function that strips item headers, skips short/empty sections, and removes page artifacts. One pass over the dataset.

## Group 4 — Moderate cleaning required

### PleIAs/SEC

**Used in:** Both plans (pretraining — 26% budget / 31% full)

This is our most important finance dataset and the one that needs the most attention. PleIAs uses EDGAR-Crawler to extract text from SEC EDGAR HTML filings. The extraction is good but SEC 10-K filings are notoriously messy documents.

**Issues present in the raw data:**

**1 — ASCII financial tables**
SEC filings contain financial statements formatted as text tables:
```
                              Year Ended December 31,
                          2023         2022         2021
                        --------     --------     --------
Net revenues          $ 394,328    $ 365,817    $ 274,515
Cost of sales           223,546      201,471      162,994
                        --------     --------     --------
Gross margin            170,782      164,346      111,521
```
These dashes, dollar signs, and spacing patterns waste merge slots during BPE tokenizer training. They should be converted to readable prose or stripped entirely.

**2 — Legal boilerplate**
Every 10-K ends with identical boilerplate: signature pages, exhibit listings, consent of accountants. Example:
```
EXHIBIT INDEX
Exhibit 21.1 — Subsidiaries of the Registrant
Exhibit 23.1 — Consent of Independent Registered Public Accounting Firm
Exhibit 31.1 — Certification of Principal Executive Officer
```
These sections are identical in structure across all filings and teach the model nothing useful.

**3 — Item headers and cross-references**
```
ITEM 1A. RISK FACTORS
See Item 7, "Management's Discussion and Analysis."
Incorporated herein by reference to Exhibit 13.
```

**4 — Cover page metadata**
SEC filings start with structured cover page data: CIK numbers, SIC codes, fiscal year end dates, filer addresses. This is not natural language.

**What to fix:**
- Strip exhibit index sections (everything after "EXHIBIT INDEX" at end of document)
- Convert or strip ASCII financial tables (strip the dashes/spacing; keep numbers as prose if possible)
- Remove cross-reference-only lines ("See Item 7", "Incorporated by reference")
- Skip cover page metadata blocks

**What to keep:**
- Risk factor text (Item 1A) — high value, natural language
- MD&A sections (Item 7) — rich financial narrative
- Business description (Item 1) — company descriptions
- Legal proceedings (Item 3) — legal language is useful

**Estimated effort:** 100–150 line cleaning function. The tricky part is the ASCII table detection (heuristic: lines with 3+ consecutive dashes or spaces + dollar signs). Worth spending time here since this dataset is 26–31% of training data.

**Verify before committing:** Inspect 10–20 random documents from the raw Parquet files before large-scale tokenization. PleIAs claims LLM-ready output but the actual quality varies by filing year and company size.

### S&P 500 Earnings Transcripts (kurry)

**Used in:** Full plan only (pretraining — ~1%)

33K earnings call transcripts from 685 companies, 2005–2025. These are spoken language converted to text. The speech-to-text quality is generally good but the transcript format introduces noise.

**Issues present:**

**1 — Speaker labels throughout**
```
OPERATOR: Thank you. Our next question comes from John Smith with Goldman Sachs.
ANALYST: Hi, good morning. Thanks for taking my question. I had two questions...
CFO: Sure, John. Great question. So if you look at our margins...
```
Speaker labels on every paragraph break the natural language flow. The model should learn what the CFO said, not the label pattern `"SPEAKER_NAME: text"`.

**2 — Call opening and closing boilerplate**
Every transcript starts and ends identically:
```
Good morning, ladies and gentlemen, and welcome to the [Company] Q3 2024 earnings call.
At this time all participants are in a listen-only mode. A question-and-answer session
will follow the formal presentation. [Standard operator script...]
```
This boilerplate is identical across thousands of calls and trains the model on useless patterns.

**3 — Operator transition phrases**
```
Thank you. Our next question comes from...
Please go ahead.
Thank you. And our next question...
```

**4 — Filler speech markers**
Spoken language converted to text sometimes preserves: `"um"`, `"uh"`, `"you know"`, repetitions.

**What to fix:**
- Strip operator lines (detect by "OPERATOR:" prefix or "Thank you. Our next question")
- Strip call opening (~first 5 lines) and closing (~last 5 lines)
- Remove speaker labels but keep the text: `"CFO: Revenue grew..." → "Revenue grew..."`
- Optionally strip filler words (low priority)

**Estimated effort:** 60–80 line cleaning function. Speaker label stripping is straightforward with regex. Boilerplate is detectable by position (first/last N lines).

## Group 5 — Heavy cleaning required

### Finance-Instruct-500k

**Used in:** Full plan only (SFT)

518K rows compiled from 37 sources. This is the most complex dataset to prepare.

**Issues:**

**1 — Train/test contamination (critical)**
Contains FinQA, TAT-QA, and ConvFinQA rows — the same benchmarks we use for evaluation. Must filter these out before training or evaluation scores are meaningless.

**2 — Format inconsistency across 37 sources**
Each of the 37 source datasets has different field names, conversation structures, and quality levels. Some use `instruction/output`, others `question/answer`, others multi-turn `messages`. Normalising all to our format requires 37 individual converters or a robust schema detector.

**3 — Quality variance**
Mixing 37 sources guarantees some low-quality rows. The dataset card says 60K+ duplicates were removed but quality filtering beyond deduplication is limited.

**4 — Language mixing**
Contains English and Chinese (ZH) examples. Our tokenizer is English-only trained. Chinese text will tokenize badly and should be filtered out.

**What to fix:**
- Filter rows where `source` field contains: `"finqa"`, `"tat-qa"`, `"convfinqa"` (eval contamination)
- Filter rows where `language == "zh"` or text contains non-ASCII majority
- Normalise all 37 source formats to our `{"role": ..., "content": ...}` structure
- Optionally: quality filter by response length (filter out < 20 token responses)

**Estimated effort:** 200–300 lines across multiple converter functions. Significant testing required to verify each source subset converts correctly. This is why the budget plan drops it in favour of Finance-Alpaca.

## Practical recommendation

For the **budget plan** — start with the three cleanest sources:
1. FineWeb-Edu (ready) + PleIAs/SEC (moderate cleaning) + OpenHermes (format extract) for pretraining
2. Finance-Alpaca (format normalise) + SmolTalk (ready) for SFT

Total cleaning code: ~250–300 lines across two scripts. Realistic to complete in a few hours before the RunPod session.

For the **full plan** — add EDGAR-Corpus (light) and Earnings Transcripts (moderate) for pretraining, and tackle Finance-Instruct-500k (heavy) for SFT only after the budget plan is validated.
